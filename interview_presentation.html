Project Overview

Goal: Build a production-ready Web Application that intelligently recommends music tailored to user preferences, matching modern platforms like Spotify.

Machine Learning (Scikit-Learn, Pandas)
Backend (Flask, SQLAlchemy, JWT)
Frontend (Vanilla JS, Plotly.js, GSAP)

1. The Data Pipeline

Data is the fuel for our engines. The project handles two main datasets:

  • Song Metadata: Audio features (danceability, energy, tempo) and genre tags.
  • User Listening History: Millions of rows mapping users to track play counts.

Preprocessing Steps:

  • Removed duplicates and handled missing values (NaNs).
  • Normalized text formats (lowercase processing).
  • Filtered the master song dataset to match tracks that actually exist in the user history to avoid cold-start issues during collaborative filtering.

2. Content-Based Filtering

Recommends songs that sound mathematically similar to the input track.

Input Song ➡ Extract Features ➡ Find Nearest Neighbors
  • Text Features (Genre/Tags): Processed using TfidfVectorizer to understand textual similarity.
  • Continuous Features (Tempo, Duration): Normalized using StandardScaler.
  • Bounded Features (Danceability 0-1): Scaled using MinMaxScaler.
  • Similarity Metric: Cosine Similarity is computed across this combined high-dimensional vector space.

3. Collaborative Filtering

Recommends songs based on "Users who listened to this also listened to that."

  • Interaction Matrix: Transformed millions of play-count records into a scipy.sparse.csr_matrix. Rows = Tracks, Columns = Users.
  • Dask for Big Data: Used dask.dataframe to powerfully process the massive user history file out-of-core (exceeding RAM limits).
  • Similarity: Calculates Item-Item Cosine Similarity between track vectors.

4. The Hybrid Engine

The true power of the platform: Combining both models to offset their individual weaknesses.

Score = (Weight_CB * Normalized_CB_Score) + (Weight_CF * Normalized_CF_Score)
  • Weights: Configured to mathematically favor Collaborative data (e.g., 70% CF, 30% CB) when community data is strong.
  • Normalization: Both score outputs are Min-Max normalized to scale (0 to 1) before blending to ensure mathematical parity.

5. Backend (Flask & SQLite)

A robust API serves the frontend and manages state.

  • Blueprints: Organized into modular routes (`routes.py`) and Auth (`auth.py`).
  • Authentication:
    • Flask-Bcrypt for hashing passwords.
    • Flask-JWT-Extended for Secure, HTTPOnly session cookies.
  • Database (SQLAlchemy): Persistent models mapping User to their custom Playlist and Liked Songs.
  • Search API: A dedicated JSON endpoint queries the pandas dataframe dynamically for global search lookups.

6. Premium UI & Frontend

A focus on aesthetics and interactive user experience.

  • Plotly.js Radar Charts: Visually renders the acoustic fingerprint of a song (Danceability vs Acousticness vs Energy).
  • External APIs: Asynchronous calls hit the iTunes Search API to fetch high-res album art and 30-second audio previews on the fly.
  • Global Audio Player: A custom-built bottom player bar manages playback state (Play/Pause/Seek/Volume) globally across the app.
  • CSS Architecture: Glassmorphism cards, dynamic grid layouts, and staggered transition animations.

7. Model Evaluation & Verification

Because this is unsupervised ranking, we evaluate differently than standard Accuracy.

  • Offline Metrics (Leave-One-Out): Hiding 20% of a user's listening history and checking:
    • Precision@K / Recall@K: Did the hidden songs appear in our Top K recommendations?
    • NDCG: Were the most relevant songs ranked at the very top?
  • Experience Metrics: Measuring Diversity (avoiding just one artist) and Novelty (surfacing hidden gems).
  • Online Metrics (A/B Testing): In production, we measure business KPIs like: Click-Through Rate (CTR) on the Play Preview button and Add-to-Playlist conversion rate.

8. Challenges Overcome

  • API Rate Limiting: Fetching 10 recommendation images concurrently caused iTunes API (HTTP 429) failures. Fix: Implemented a synchronous delay() queue to stagger requests by 200ms.
  • Audio State Syncing: Clicking a play button on a card didn't update the global player bar. Fix: Built a global state tracker referencing the active DOM element to instantly toggle ⏸ and ▶ everywhere.
  • Data Science Code Quality: Refactored raw flat scripts into an industry-standard Cookiecutter data-science directory structure (src/models, notebooks/, data/processed) separating concerns perfectly.

8. Conclusion & Future Scope

MuseRec successfully demonstrates the intersection of full-stack engineering and machine learning.
